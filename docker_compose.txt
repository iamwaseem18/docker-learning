## What is Docker Compose?

Docker Compose is a tool used to define and run **multi-container Docker applications** using a single YAML file (`docker-compose.yml`).

Instead of running multiple `docker run` commands, you define everything once and start all services together.

---

## Why Docker Compose?

Managing multiple containers manually becomes messy.

Example:

* Web app
* Database
* Cache
* Worker service

Without Compose → multiple commands
With Compose → one command:

```bash
docker compose up
```

---

## Running Individual Containers (Without Compose)

```bash
docker run mmunshad/simple-webapp
docker run mongodb
docker run redis:alpine
docker run ansible
```

Problem:

* Hard to manage
* No easy networking
* No dependency handling

---

## Basic Docker Compose File

```yaml
version: '3.8'

services:
  web:
    image: mmunshad/simple-webapp

  database:
    image: mongodb

  messaging:
    image: redis:alpine

  orchestration:
    image: ansible
```

Run:

```bash
docker compose up
```

---

## Key Concepts

### 1. services

Defines containers in your application.

### 2. image

Specifies which Docker image to use.

### 3. ports

Maps host → container ports.

```yaml
ports:
  - "5000:80"
```

### 4. depends_on

Controls startup order.

```yaml
depends_on:
  - redis
```

## Docker Run vs Docker Compose

### Docker Run (Manual)

```bash
docker run -d --name=redis redis
docker run -d --name=db postgres
docker run -d --name=vote -p 5000:80 voting-app
docker run -d --name=result -p 5001:80 result-app
docker run -d --name=worker worker
```

Issues:

* No automatic networking
* Hard to manage dependencies

---

### Docker Compose Equivalent

```yaml
version: '3.8'

services:
  redis:
    image: redis

  db:
    image: postgres:9.4

  vote:
    image: voting-app
    ports:
      - "5000:80"
    depends_on:
      - redis

  result:
    image: result-app
    ports:
      - "5001:80"
    depends_on:
      - db

  worker:
    image: worker
    depends_on:
      - redis
      - db
```

## Build vs Image

Use `image` if pulling from Docker Hub:

```yaml
image: nginx
```

Use `build` if building locally:

```yaml
build: ./app
```

## Docker Compose File Versions

### Version 1

* No `services` section
* Uses `links` (deprecated)

### Version 2

* Introduced `services`
* Added `depends_on`

### Version 3

* Supports Docker Swarm
* Better networking

---

## Networking in Docker Compose

By default:

* All services can talk using service names

Example:

```yaml
redis:
  image: redis
```

Other containers connect using:

```
host = "redis"
```

---

## Custom Networks

```yaml
version: '3.8'

services:
  web:
    image: voting-app
    networks:
      - front-end
      - back-end

  redis:
    image: redis
    networks:
      - back-end

networks:
  front-end:
  back-end:
```

### Why use this?

* Separate public and internal traffic
* Improve security

---

## Deprecated: Links

Old method:

```yaml
links:
  - redis
```

Modern replacement:

* Use default networking (automatic)

---

## Common Commands

Start services:

```bash
docker compose up
```

Run in background:

```bash
docker compose up -d
```

Stop services:

```bash
docker compose down
```

View logs:

```bash
docker compose logs
```

---

## Key Interview Points

* Docker Compose = multi-container management
* Uses YAML file
* Services communicate via service names
* `depends_on` controls startup order
* Better than multiple `docker run` commands

---

## Final Summary

Docker Compose simplifies:

* Running multiple containers
* Networking between services
* Managing dependencies
* Application deployment

One file → full application stack
